# Daybook — Spec

## Overview
A simple daily accounting PWA for personal use on iPhone. Track a running cash/account balance day by day with credit and debit transactions across multiple named ledgers. Data is stored in Supabase (Postgres) with Google OAuth — any Google account can sign in and gets its own isolated data. A serverless function on Vercel handles email report delivery.

**Production URL:** https://daybook-lyart.vercel.app
**Dev/Preview URL:** https://daybook-dev.vercel.app
**Repo:** https://github.com/Rishi-Kumar/daybook.git

## Auth

Google OAuth via Supabase (PKCE flow). On sign-in, Supabase redirects back to `window.location.origin`. Session is persisted in localStorage by the Supabase client and auto-attached to all requests as a JWT.

`App.jsx` uses `supabase.auth.onAuthStateChange` to track session state:
- `undefined` → loading (render nothing)
- `null` → unauthenticated → show `AuthScreen`
- object → authenticated → run ledger init, show main UI

Sign-out is accessible via the header of the Transactions screen.

## Data Model

All tables live in Supabase Postgres. Row Level Security (RLS) is enabled on all tables, scoped to `auth.uid() = user_id`.

### `ledgers`
| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | Generated by client via `uuid` package |
| `user_id` | uuid | Auth user, enforced by RLS |
| `name` | text | Display name (e.g. "Cash") |
| `opening_balance` | numeric(15,2) | Balance at the start of this ledger |
| `setup_date` | date | The date the ledger was created |
| `created_at` | bigint | Unix timestamp ms (used for ordering) |

### `transactions`
| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | Generated by client via `uuid` package |
| `user_id` | uuid | Auth user, enforced by RLS |
| `ledger_id` | uuid (FK → ledgers) | ON DELETE CASCADE |
| `date` | date | The day this transaction belongs to |
| `type` | text | `'credit'` or `'debit'` |
| `amount` | numeric(15,2) | Positive for credit, negative for debit |
| `particulars` | text | Free-text description |
| `created_at` | bigint | Unix timestamp ms (for ordering within a day) |

**Index:** `(user_id, ledger_id, date)` for efficient range queries.

### `user_settings`
| Column | Type | Description |
|--------|------|-------------|
| `user_id` | uuid (PK, part) | Auth user, enforced by RLS |
| `key` | text (PK, part) | Setting name (e.g. `activeLedgerId`) |
| `value` | text | Setting value |

### Client-side cache (localStorage)
A full snapshot of ledgers + transactions is written to `localStorage` under `daybook_cache` after each successful Supabase load. A write queue of pending operations is stored under `daybook_queue`. User settings are mirrored under `daybook_setting_<key>`. This enables offline read and write — see Offline Support below.

## Business Logic

### Opening balance for a given day
- Sum of `opening_balance` plus all transaction amounts before that date
- `getTransactionGroupsForRange(from, to, ledgerId)` — 3 parallel queries: ledger, in-range transactions, prior transactions. Computes running opening/closing balances in JS.
- `getAllLedgersWithBalances()` — 2 parallel queries: all ledgers, all transactions. Sums per ledger in JS.

### Closing balance for a given day
```
closingBalance = openingBalance + sum(signed amounts for that day)
```

### Transaction sort order within a day
Credits first, then debits. Within each group, sorted by `createdAt` ascending (order added). Comparator: `compareTx(a, b)` in `utils.js`.

### Closing balance row display
- Shown at month boundaries and at the final date in the query range
- Month-end closing balance rows include an "End of MMM YYYY" label (except the final row)
- Closing balance text is green if positive, red if negative

## Offline Support

On app init, after fetching from Supabase, all ledgers and transactions are written as a JSON snapshot to `localStorage` (`daybook_cache`).

**Reads when offline:** All DB read functions check `navigator.onLine` first. If offline, they derive results from the cache using the same JS logic as the online path.

**Writes when offline:** All write functions update the cache immediately (optimistic), then check `navigator.onLine`. If offline, the operation is appended to `daybook_queue` and returns. If online, it goes to Supabase directly.

**Reconnect sync:** A `window.addEventListener('online', flushQueue)` is registered at module load in `db.js`. On reconnect, queued operations are replayed against Supabase in order. On full success the queue is cleared and the cache is refreshed from the server. First failure stops the flush — it retries on the next reconnect.

**Settings offline:** `activeLedgerId` is mirrored to `localStorage` so the app can initialise without a network request.

**Network indicator:** A 7px dot in both screen headers — green (`var(--credit)`) when online, red (`var(--debit)`) when offline.

## Screens

### 1. Auth Screen
- Shown when unauthenticated (`session === null`)
- App name + tagline + "Sign in with Google" button
- Triggers Supabase Google OAuth PKCE redirect

### 2. Setup Screen (first login only)
- Shown when authenticated but no ledgers exist
- Ledger Name input (defaults to "Cash")
- Opening Balance input with +/− sign toggle (allows negative balances)
- "Get Started" button creates the ledger and navigates to the Transactions tab

### 3. Transactions Screen (Transactions tab)
- Header: app name + ledger name, network dot, email icon, sign-out icon
  - Sign-out shows an inline confirmation ("Sign out? / Cancel / Yes") before calling `supabase.auth.signOut()`
  - Email button disabled while loading
- Date range picker: From / To date inputs, defaults to last 7 days
- Transactions grouped by day in ascending date order
- Each day group:
  - Sticky date header
  - Opening Balance row (shown on the first day of the range)
  - All transactions for that day (credits first, then debits)
  - Closing Balance row (shown at month-end and the last day in range)
- Each transaction row:
  - +/− sign badge (colour-coded green/red)
  - Particulars
  - Amount right-aligned
  - Long-press (500 ms) or right-click → opens Edit sheet
- Empty state when no transactions in range

### 4. Add / Edit Transaction Bottom Sheet
- Slide-up modal with drag handle; triggered by the center `+` nav button (add) or long-press/right-click on a row (edit)
- × close button in header; tapping the backdrop also dismisses
- Fields in order:
  1. Date (date input, defaults to today; capped at today — no future dates)
  2. Particulars (text input, auto-focused on open; Enter submits)
  3. +/− sign toggle (pill selector)
  4. Amount (numeric input; Enter submits)
  5. Ledger picker (only shown when 2+ ledgers exist)
- Save button label: "Save" (add mode) or "Update" (edit mode)
- Delete button appears in edit mode only, with a confirmation dialog
- Validates amount > 0; shows inline error if invalid

### 5. Ledgers Screen (Ledgers tab)
- Header: app name, network dot, "New" button
- Cards listing all ledgers, each showing:
  - Ledger name
  - Active indicator dot (if currently selected)
  - Current balance (negative shown in red)
- Tapping a card sets it as the active ledger and switches to the Transactions tab
- Long-press (500 ms) or right-click on a card → opens Edit Ledger sheet

### 6. Add / Edit Ledger Bottom Sheet
- Slide-up modal with drag handle
- Fields: Name (required, auto-focused), Opening Balance with +/− toggle
- "Create Ledger" (add) or "Save Changes" (edit) button
- Delete button in edit mode with confirmation (cascades all transactions via Postgres ON DELETE CASCADE)

### 7. Email Report
- Triggered by the envelope icon in the Transactions screen header
- Slide-up bottom sheet:
  - Email input pre-filled with last-used address (persisted in `user_settings` as `reportEmail`)
  - "Send Report" button — disabled while empty or sending
  - Success state: green checkmark + "Report sent to {address}" + Done button
  - Error state: inline error message
- On send: POSTs `{ to, fromDate, toDate }` to `/api/send-report` with `Authorization: Bearer <session.access_token>`
- `/api/send-report` serverless function:
  - Verifies bearer token via Supabase admin client; returns 401 if invalid
  - Fetches the user's ledgers + transactions for the date range server-side (using `buildLedgerGroups` from `src/utils.js`)
  - Generates HTML email body
  - Generates a PDF attachment via jsPDF + jspdf-autotable:
    - One section per ledger (only ledgers with transactions in range); each after the first starts on a new page
    - Header: "Daybook Report", date range, generated timestamp
    - Per-ledger section: ledger name heading, table with Date | Particulars | Credit | Debit columns
    - Opening Balance and Closing Balance rows highlighted in light gray + bold
    - Credit amounts in green (#158b3d), debit amounts in red (#b91c1c)
    - Page break guard: if `y > 265 mm`, new page added
  - Sends via Nodemailer + Gmail SMTP
  - Returns `{ ok: true }` on success, `{ error: string }` on failure

## Currency & Formatting
- All amounts in INR
- No currency symbol displayed anywhere (neither in-app nor in reports)
- No `+`/`−` prefix on amounts in history; colour conveys direction
- Format: Indian numbering system with 2 decimal places
  - `Intl.NumberFormat('en-IN', { minimumFractionDigits: 2, maximumFractionDigits: 2 })`
- Positive amounts: green (`var(--credit)`)
- Negative / debit amounts: red (`var(--debit)`)

## Constraints
- Mobile-first: designed for 375px–430px (iPhone)
- Auth required: data is scoped per Google account via Supabase RLS
- Offline read+write supported via localStorage cache and write queue; first load requires connectivity
- Multiple ledgers, single currency (INR)
